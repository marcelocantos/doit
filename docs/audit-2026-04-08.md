## Detailed findings

### Phase 1: Code quality

#### 1.1 Architecture and design

**Info** The codebase is well-structured along clear module boundaries: `cap/` for capabilities, `pipeline/` for parsing, `policy/` for the three-level engine, `engine/` for the public API, `mcptools/` for MCP integration. No circular dependencies detected. The v2 direction (MCP server, `sh -c` execution, Starlark rules) is partially implemented alongside the v1 CLI path, which is appropriate for a transitional codebase.

**Info** The dual-path architecture (CLI via `cmd/doit` and MCP via `cmd/doit-mcp`) shares logic well through the `engine` package. The `engine.Engine` type cleanly composes config, registry, audit, and all three policy levels.

#### 1.2 Code correctness

1. **#2** (High) `resolveError` in `internal/cli/run.go:106` is dead code. It was the pre-`sh -c` error resolver. `runShell` handles its own error resolution inline. The function is only referenced in tests (`cli_test.go`). Remove both the function and its tests.

2. **#12** (Low) `audit.Tail` (`verify.go:55`) silently skips entries that fail `json.Unmarshal` (line 72: `continue`). A corrupted entry in the middle could make `Tail` return fewer entries than requested without any error indication. Consider returning an error or logging a warning.

3. **#13** (Low) `policy.TokenStore` issues tokens but only purges via an explicit `Purge()` call. In the long-running MCP server (`doit-mcp`), there is no goroutine or ticker calling `Purge()`. Expired tokens accumulate in memory indefinitely. For a lightly-used server this is negligible, but a periodic cleanup goroutine would be more correct.

**Info** Error handling is generally thorough throughout. Audit logging failures are correctly handled as best-effort (`_ = logger.Log(...)`), and the main entry point propagates exit codes cleanly via `ExitError`.

#### 1.3 Code hygiene

4. **#3** (High) `hasAnyFlag` exists in two separate implementations:
   - `internal/rules/rules.go:52` (unexported `hasAnyFlag`)
   - `internal/policy/level1.go:278` (exported `HasAnyFlag`)

   Both are identical in logic. `level2.go` imports the exported one. The `rules` package uses its own copy. This is a maintenance hazard -- a bug fix in one won't reach the other. The `rules` package should import `HasAnyFlag` from `policy`, or the shared function should live in a common utility package.

5. **#4** (High) `DaemonConfig` in `config.go:49-55` and `IdleTimeoutDuration()` are unused -- the daemon was removed in favour of the MCP server. The help text (`cli/help.go:54`) still advertises `doit --daemon`. These should be cleaned up.

6. **#2** (see above) `resolveError` is dead code.

**Info** No significant duplication beyond `hasAnyFlag`. Code complexity is well within bounds -- the longest function is `engine.Execute` at ~75 lines, which is reasonable for its orchestration role.

### Phase 2: Security

#### 2.1 Input handling

**Info** The primary security model is sound: commands are parsed and validated through the capability registry and policy chain before execution. The `sh -c` execution path delegates shell parsing to the system shell, which is the correct design for a command broker.

**Info** The Starlark sandbox is properly configured -- `loader.go` uses `starlark.ExecFileOptions` with no filesystem access beyond what the rule file itself defines. The `json` module is the only predeclared builtin.

#### 2.2 Secrets and credentials

**Info** No hardcoded secrets, API keys, or credentials found in the source tree. The LLM client strips `CLAUDECODE*` environment variables before calling `claude` (`llm/client.go:60`), which prevents credential leakage to the subprocess.

#### 2.3 Cryptography

**Info** SHA-256 is used appropriately for the audit hash chain (integrity, not secrecy). Token generation uses `crypto/rand` with 128-bit entropy, which is adequate for time-limited single-use tokens.

#### 2.4 Access and exposure

**Info** No network listeners. The MCP server uses stdio transport only. No CORS, no HTTP endpoints.

### Phase 4: Performance

**Info** No significant performance concerns for a command broker. The audit log reads the entire file into memory for verification and querying (`os.ReadFile`), but audit logs are unlikely to exceed a few MB in practice. If they did, `audit.max_size_mb` (config field) is defined but not enforced (see finding #6).

7. **#6** (Medium) `config.AuditConfig.MaxSizeMB` is defined and defaulted to 100 in `DefaultConfig()`, but no code checks or enforces this limit. The audit log can grow unboundedly, and `Verify`/`Query`/`Tail` all read the entire file into memory.

### Phase 5: Build, CI/CD, and infrastructure

#### 5.1 Build system

**Info** `Makefile` is clean and correct. Parallel builds are enabled via `MAKEFLAGS`. The generated file dependency (`internal/cli/help_agent.md`) is properly declared. Both binaries (`doit`, `doit-mcp`) build successfully.

#### 5.2 CI/CD

8. **#1** (High) The only CI workflow is `release.yml`, which runs on release publication. There is **no CI for push or pull requests** -- no automated tests, vet, or linting on PRs. This means the default branch has no CI protection. A `.github/workflows/ci.yml` running `make test` and `make vet` on push/PR to master is needed.

9. **#10** (Medium) The release workflow builds only `doit` but not `doit-mcp`. Since `doit-mcp` is the MCP server binary (the primary interface for the v2 architecture), it should also be built and published in releases.

**Info** Merge settings are correctly configured: squash-only, PR title as commit message, delete-branch-on-merge.

#### 5.3 Repository hygiene

**Info** `.gitignore` covers the essentials (`bin/`, `.DS_Store`, editor files, `cover.out`, generated `help_agent.md`). No sensitive files in the repo.

10. **#11** (Low) Minor: `.gitignore` could additionally cover `node_modules/`, `__pycache__/`, and `*.log` as defensive entries, though these are unlikely to appear in a Go project.

## Filtered out

The following potential findings were investigated and determined to be non-issues:

- **`cwd, _ := os.Getwd()` ignoring error in main.go**: Acceptable -- if `Getwd` fails, empty string is fine (child process uses its own cwd).
- **`data, _ := json.Marshal(e)` in `computeHash`**: The entry is a struct of basic types; `json.Marshal` cannot fail here.
- **Starlark `Rule.Thread` reuse**: Starlark threads are not goroutine-safe, but rules are evaluated sequentially in the L1 engine. No concurrent access.
- **`os.UserHomeDir` errors ignored in `DefaultConfig`**: Returns empty string, which results in a path like `/.local/share/doit/...`. The subsequent `MkdirAll` will fail with a clear error if this is wrong.
